# Practice 08 — 부서별 통합 파이프라인: Plan 모드 + 멀티파일

## 역할
당신은 실습 총괄 강사입니다.
이 실습은 지금까지 배운 도구를 한 번에 묶는 시간입니다.
개별 파일 분석이 아니라, 여러 파일을 연결한 부서별 통합 흐름을 설계하고 실행하게 만드세요.

## 실습 목표
- Plan 모드의 필요성을 이해한다.
- 부서별 멀티파일 분석 파이프라인을 실행한다.
- 여러 데이터 소스를 연결해 하나의 산출물로 만든다.
- 배포용 결과물을 준비한다.

## 산출물
- 부서별 통합 분석 보고서 1개 (A: `engineering_review.md`, B: `strategy_review.md`, C: `manufacturing_review.md`, D: `executive_budget_review.md`)

## 성공 기준
- 각 그룹이 Plan 모드로 작업을 시작했다.
- 그룹별 통합 분석 프롬프트를 실행했다.
- 결과물이 단일 파일로 정리되었다.
- 공유 가능한 산출물 초안이 생성되었다.

## 선행 조건
- Practice 07 완료 (3층 비교 + 프롬프트 4법칙)

## 오프닝 스크립트
"지금까지는 파일 하나를 잘 다뤘다면, 이제는 여러 파일을 연결해 하나의 결과물로 만들어봅니다. 여기서부터는 바로 실행보다 계획이 더 중요합니다."

## 진행 순서

### Step 1 — Plan 모드 설명
강사 멘트:
"작업이 커질수록 바로 시키는 것보다 먼저 계획을 보는 편이 안전합니다."

설명 포인트:
- 읽을 파일이 여러 개일 때
- 중간 산출물이 필요할 때
- 최종 결과 형식을 미리 합의해야 할 때

**ACTION**
- 계획 확인 → 수정 → 승인 흐름을 보여준다.

**USER**
- Plan 모드의 장점을 한 줄로 적는다.

**STOP**
- "복잡할수록 계획이 먼저"라는 감각을 잡았는지 확인

### Step 2 — 부서별 통합 분석

**A그룹 — R&D 통합 분석**
```
Plan 모드 ON → 입력:
"@practice-data/A-RnD/result.csv 와 @practice-data/A-RnD/battery_log.csv 를 함께 분석해서:
 1. CAE 불합격 부품 목록 (응력 500MPa 초과)
 2. 배터리 이상 이벤트 (전압 3V 미만 OR 온도 45도 초과)
 3. 두 데이터의 공통 패턴
 를 engineering_review.md로 저장해줘."
```

**B그룹 — 전략 통합 분석**
```
Plan 모드 ON → 입력:
"@practice-data/B-전략/market_reports/ 와 @practice-data/B-전략/kpi_data.csv 를 통합 분석해서:
 1. 시장 트렌드가 KPI 달성에 미치는 영향
 2. 목표 미달 KPI 항목별 시장 환경 요인
 3. 다음 분기 전략 권고사항 3가지
 를 strategy_review.md로 작성해줘."
```

**C그룹 — 제조 통합 분석**
```
Plan 모드 ON → 입력:
"@practice-data/C-제조/defect_log.csv 와 @practice-data/C-제조/production.csv 를 함께 분석해서:
 1. 생산량 대비 불량률 추이
 2. 생산량 급증 시점과 불량 증가 상관관계
 3. 불량 감소를 위한 개선 제안
 를 manufacturing_review.md로 만들어줘.
 @practice-data/C-제조/mold_history.csv 에서 금형 교체 일정도 포함해줘."
```

**D그룹 — 경영 통합 분석**
```
Plan 모드 ON → 입력:
"@practice-data/D-경영/budget_2026.csv, @practice-data/D-경영/actual_Q1.csv, @practice-data/D-경영/purchase_history.csv 를 통합 분석해서:
 1. 예산 달성률 하위 항목과 구매 지출 상관관계
 2. 반복 구매 품목의 가격 최적화 기회
 3. Q2 예산 재배분 제안
 를 executive_budget_review.md로 만들어줘."
```

**ACTION**
- 각 그룹이 Plan 모드로 시작하게 한다.
- 계획을 먼저 읽고 빠진 항목이 있으면 보완한다.

**USER**
- 멀티파일 입력을 명시한다.
- 최종 산출물 이름을 지정한다.
- 결과물을 저장한다.

**STOP**
- 각 그룹이 계획을 거쳤는지 확인

### Step 3 — 산출물 보완
보완 질문 예시:
- `이 결과를 팀장 보고용으로 더 짧게 요약해줘`
- `표 중심으로 다시 정리해줘`
- `HTML 대시보드로 옮기기 좋게 섹션을 재구성해줘`

**ACTION**
- 공유/배포에 맞는 형태로 결과를 다듬게 한다.

**USER**
- 최종 버전을 하나로 고정한다.

**STOP**
- 최종본을 명확히 지정

> 강사 멘트: "CLAUDE.md가 있으니까 제가 규칙을 말 안 해도 AI가 알아서 적용했죠? 이게 구조의 힘입니다."

**클로징:**
```
"저장해줘" → commit (통합분석 결과물 포함)
"올려줘" → push
```

## 트러블슈팅
- 멀티파일 결과가 산만함 → 산출물 섹션을 먼저 지정
- 계획이 너무 길어짐 → 목적과 출력 형식을 단순화
- 숫자 누락 → 입력 파일명과 필수 표 항목을 다시 명시

## 강사 운영 팁
- 결과물 품질보다 파이프라인 경험이 중요합니다.
- 한 번에 완벽한 보고서를 만들려 하지 말고, 계획-실행-보완의 흐름을 체험하게 하세요.
